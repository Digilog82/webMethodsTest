Socket Log > Local Server Queue Info Screen
Screen Heading: IS Administrator > Solutions > Socket Admin > Socket Log > Local Server Queue Info
Local Socket Server Port로 수신된 데이터를 UM에 전송하는 경우 아직 처리되지 않은 데이터를 조회하고 Skip 처리가 가능하도록 설정하는 메뉴이다.
  • Socket Log > Local Server Queue Info 클릭 : 조회화면으로 이동한다.
  • Search 클릭 : 조회조건에 해당하는 목록을 조회한다.
  • Update Skip Status 클릭 : 선택한 전문 데이터에 대해서 처리되지 않도록 Skip Status를 Yes로 업데이트 한다.
  • Queue Send Time 링크 클릭 : Request Data 상세조회 화면으로 이동한다.
Search Screen
Note:  
  • Basic Info > Custom Variable 메뉴의 ESB.SOCKET.QINFO.SAVEYN의 Value가 true인 경우에만 로그파일로 저장된다.
  • 로그파일 Naming Rule은 아래와 같다.
    Port Number + "$" + Queue/Topic Name + "$" + Send Time + "$" + interfaceID + "$" + 전문ID + "$" + serverIP + "$" + UUID
  • 파일 저장 위치
    SAG_Directory/CUSTOM_CONFIG/socket/qinfos
    운영서버의 경우 반드시 NAS 등 공유 디렉토리로 설정해야 한다.
  • Doc ID 링크를 클릭하면 해당 Transaction에 대한 상세 데이터를 조회하는 화면으로 이동한다.
    상세조회 화면에서는 Transaction의 기본정보, 전문의 Schema 정의, 전문의 각 필드의 데이터를 조회할 수 있고 해당 전문 한 건에 대해서 Update Skip Status 처리를 할 수 있다.
  • Local Socket Server Port로부터 데이터를 수신하는 Listener에서 수신한 데이터를 UM에 전송하는 경우에 UM에 전송한 데이터를 똑같이 파일 시스템에 저장한다. UM에 전송한 데이터를 처리하기 위해서 Trigger가 동작하고 Trigger에서 호출하는 Distributer에서 해당 데이터를 처리하고 나면 Listener에서 저장한 파일을 Distributer에서 삭제한다.
    만약, 순차처리를 하는 경우에는 처리 시간이 오래 걸리는 데이터가 있을 경우 그 이후의 모든 데이터가 처리되지 않고 UM Queue에 대기하게 된다. 이처럼 오랫동안 처리되지 않고 있는 전문 데이터를 이 메뉴에서 조회할 수 있고 특정 전문을 선택해서 Distributer에서 전문처리하는 것을 Skip 하도록 할 수 있다.
  • Skip Status를 Yes로 업데이트 하면 파일로 저장되어 있던 전문 데이터가 사라지고 Skip Status를 No로 복구할 수 없다.
  • Skip Status를 Yes로 업데이트 한 후에 해당 전문을 처리하는 서비스가 실행 중이면 서비스의 Thread ID를 이용해서 Thread를 Kill 시키게 된다. 이 기능이 정상적으로 동작하려면 전문을 처리하는 서비스가 실행될 때 서비스의 Thread ID를 구하고 Thread ID 정보를 IS 메모리에 저장하는 로직을 구현해야 한다.
  • Update Skip Status 버튼을 클릭하기 위해서는 적어도 하나 이상 체크박스를 선택해야 한다.

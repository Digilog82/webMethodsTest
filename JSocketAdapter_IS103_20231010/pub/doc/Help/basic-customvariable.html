Basic Info > Custom Variable Screen
Screen Heading: IS Administrator > Solutions > Socket Admin > Basic Info > Custom Variable
JSocketAdapter를 구현하는데 필요한 Values에 대해서 소스 안에서 Set Value 하는 것을 피하고 소스 수정 없이 Values를 변경하면 바로 반영될 수
있도록 하기 위해서 Name/Value 쌍으로 Variables를 [등록/수정/삭제/조회] 하는 메뉴이다.
JSocketAdapter Package를 최초에 Install 하면 Custom Variable 기본값들이 자동으로 등록된다.
JSocketAdapter Package Install 후 가장 먼저 해야할 일은 System 상황에 맞게 적당한 Value로 수정하는 것이다.
다음과 같은 Variable Name에 대해서는 특히 주의해서 수정해야 한다.
- ESB.CONFIG.COMPANY.NAME(회사명)
- ESB.CONFIG.EMAIL.HOST(Email Host Name or IP)
- ESB.CONFIG.INCHARGE.EMAIL(ESB 관리자 이메일)
- ESB.CONFIG.MULTIREMOTE.ALIAS(동기화 처리할 서버가 Clustering 상관 없이 두 개 이상인 경우 Remote Server Alias 목록)
- ESB.CONFIG.OTHER.SERVERNAME(다른 운영서버의 Host Name)
- ESB.CONFIG.REMOTE.ALIAS(다른 운영서버의 Remote Server Alias)
- ESB.CONFIG.REMOTE.DEPCONFIG(Deploy 가능한 Remote Server Alias 목록)
- ESB.CONFIG.REMOTE.IP(다른 운영서버의 Remote Server IP)
- ESB.CONFIG.SERVER.TYPE(ESB 서버 구분)

Custom Variable의 Value 수정 순서(수정이 필요 없는 경우에는 Skip)
1. ESB.CONFIG.MULTIREMOTE.ALIAS
2. ESB.CONFIG.SERVER.TYPE
3. ESB.CONFIG.REMOTE.ALIAS
4. ESB.CONFIG.REMOTE.IP
5. ESB.CONFIG.OTHER.SERVERNAME
6. 나머지는 순서 상관 없음
1, 2, 3번이 먼저 설정이 되어야 다른 운영서버와 동기화 처리가 된다.

Socket Server에 대한 구성을 물리적으로 Gateway Server(Socket Port Open, Data Listening)와 Main Server(Data Processing)로 구분해서 운영하는 경우에는 다음과 같은 Variable Name에 대해서 적당하게 설정해야 한다.
Gateway Server에서의 설정
- ESB.CONFIG.MULTIREMOTE.ALIAS
  자기 자신을 제외한 Gateway Server와 Main Server에 대한 Remote Server Alias를 '/'로 구분해서 설정
- ESB.CONFIG.MASERVER.ALIAS
  Main Server에 대한 Remote Server Alias를 '/'로 구분해서 설정
  만약 Main Server에 대한 L4 스위치가 있는 경우에는 L4 스위치에 대한 Remote Server Alias 하나만 설정
- ESB.CONFIG.SERVER.ROLE = GW
- ESB.SOCKET.DOCINFO.MULTISYNCHYN = true
Main Server에서의 설정
- ESB.CONFIG.MULTIREMOTE.ALIAS
  자기 자신을 제외한 Main Server와 Gateway Server에 대한 Remote Server Alias를 '/'로 구분해서 설정
- ESB.CONFIG.GWSERVER.ALIAS
  Gateway Server에 대한 Remote Server Alias를 '/'로 구분해서 설정
  만약 Gateway Server에 대한 L4 스위치가 있는 경우에는 L4 스위치에 대한 Remote Server Alias 하나만
  설정
- ESB.CONFIG.SERVER.ROLE = MA
- ESB.SOCKET.DOCINFO.MULTISYNCHYN = true
  • Basic Info > Custom Variable 클릭 : 목록을 조회한다.
  • Create New Custom Variable 클릭 : 신규정보 등록화면으로 이동한다.
  • Edit 아이콘 클릭 : 상세조회 및 수정화면으로 이동한다.
  • Delete 아이콘 클릭 : 등록정보를 삭제한다.

Note:  
  • 기본적으로 등록되어 있는 Variables는 삭제하거나 Name을 수정하지 않도록 한다. JSocketAdapter 내에서 Variable Name을 사용하는 것이 있다.
  • Variable Name은 기존에 등록되지 않은 Unique 한 값으로 해야 한다.
  • 모든 Field에 대해서는 필수로 입력한다.
  • Variable Name은 직접 입력할 수도 있고 Custom Variable Value에 등록되어 있는 Variable Name을 선택해서 입력할 수도 있다.
  • Variable Name을 직접 입력했는데 Custom Variable Value에 등록되어 있는 Variable Name과 동일할 경우에는 자동으로 Custom Variable Value에 등록되어 있는 Values 중에서 선택할 수 있도록 된다.
  • Value를 여러 개를 등록해야 하는 경우에는 '/' 를 구분자로 해서 여러 Value를 입력한다.
    예를 들어서, ESB.CONFIG.INCHARGE.EMAIL 에 대한 Value를 여러 개 등록하는 경우 다음과 같이 입력한다.
    aaa@nblee.com/bbb@nblee.com/ccc@nblee.com
  • Custom Variable에 대한 [등록/수정/삭제] 정보는 Integration Server Memory에도 똑같이 동기화 된다.
    Integration Server Memory에 저장된 정보는 IS Memory Object > Custom Variable 메뉴에서 확인 가능하다.
  • 운영서버가 두 대인 경우 한 쪽 운영서버에서 [등록/수정/삭제] 하면 다른 쪽 운영서버에서도 자동으로 [등록/수정/삭제] 된다.
  • Config File : SAG_Directory/CUSTOM_CONFIG/socket/customVariableConfig.xml
  • Integration Server를 Restart 하거나 JSocketAdapter Package를 Reload 하면 StartUp 서비스에 의해서 Config File의 정보를 Integration Server Memory에 동기화 시킨다.
  • Variable의 Value를 수정하면 JSocketAdapter, JSocketSamples Package의 Flow Service는 소스 수정 없이 바로 적용이 가능하도록 되어 있다.
  • ESB.CONFIG.AS400.ASYNCHONLINEYN
    AS400 시스템이 존재하면서 Data Queue를 이용해서 Asynch Online 방식의 Socket 인터페이스를 위해서 ESB와 연계하는지 여부를 설정한다.
    true : ESB와 연계함, false : ESB와 연계하지 않음
    Basic Info > Business Name 메뉴에서 Asynch Online Socket YN 필드가 'Yes' 인 Business Name이 있는 경우에는 왼쪽 메뉴에 "ProviderSystemName Online" 이라는 메뉴가 Display 된다.
    그러면서 ESB.CONFIG.AS400.ASYNCHONLINEYN의 Value가 true인 경우에는 "ProviderSystemName Online" 메뉴 아래에 "Data Q Listener" 라는 메뉴가 Display 된다.
  • ESB.CONFIG.AS400.CLUSTERPATH
    AS400 Data Q Listener Clustering을 위한 임시파일 저장위치를 설정한다.
    AS400 시스템이 존재하면서 Data Queue를 이용해서 ESB와 연계하는 경우 ESB 시스템에는 Data Queue Listener를 등록한다.
    ESB 운영서버가 두 대인 경우 Data Queue Listener를 양 쪽 서버에서 모두 Enable 해서 사용할 수 없고 한 쪽 서버에서만 Enable 해서 사용해야 한다.
    이러한 제품의 한계를 극복하기 위해서 JSocketAdapter에서는 Data Queue Listener를 양 쪽 서버에서 모두 Enable 해서 사용하도록 할 수 있다.
    운영서버이고 운영서버가 이중화 되어 있으면서 두 서버에서 공유할 수 있는 파일 시스템이 있는 경우에만 설정한다.
  • ESB.CONFIG.AS400.CONFIGPATH
    AS400 Data Q Listener 관리 XML 파일 저장위치를 설정한다.
    AS400 시스템이 존재하면서 Data Queue를 이용해서 ESB와 연계하는 경우 ESB 시스템에는 Data Queue Listener를 등록한다.
    IS Administrator > Adapters > AS/400 Adapter 메뉴에서 등록한 Data Queue Listener 중에서 ESB 시스템에서 관리할 Listener를 등록하는데 이 등록한 정보를 XML 파일로 관리한다.
    이 Variable에 대해서는 Value를 수정할 수 없다.
  • ESB.CONFIG.AS400.DATAQUSEYN
    AS400 시스템이 존재하면서 Data Queue를 이용해서 ESB와 연계하는지 여부를 설정한다.
    true : ESB와 연계함, false : ESB와 연계하지 않음
    ESB.CONFIG.AS400.DATAQUSEYN의 Value가 true인 경우에는 왼쪽 메뉴에 "AS/400" 이라는 메뉴가 Display 된다.
  • ESB.CONFIG.AS400.LISTENERHC
    AS400 Data Queue Listener에 대한 Health Check 수행 여부를 설정한다.
    AS400 시스템이 존재하면서 Data Queue를 이용해서 ESB와 연계하는 경우 ESB 시스템에는 Data Queue Listener를 등록한다.
    이 Data Queue Listener는 AS400 시스템과 ESB 시스템 사이에 Online Socket으로 연결되어 있는 것으로서 일정시간 동안 데이터 송수신이 없을 경우 Network Level에서 임의로 연결을 끊을 수 있다.
    JSocketAdapter에서는 Data Queue Listener에 대한 연결이 끊어지지 않게 하기 위한 기능을 제공한다.
    ESB.CONFIG.AS400.LISTENERHC의 Value는 JSocketAdapter에서 제공하는 기능이 수행될 때 변경되는 것으로서 Basic Info > Custom Variable 메뉴에서 임의로 Value를 수정하지 않도록 한다.
  • ESB.CONFIG.AUDITLOG.CMENUDISPLAY
    Socket Log > Remote Server Audit Log 메뉴의 Display 여부를 설정한다.
    true : Display 함
    false : Display 하지 않음
    Audit Log 건수가 많을 경우 조회할 때 서버에 부하를 주기 때문에 Audit Log 건수에 따라서 Display 여부를 결정한다.
  • ESB.CONFIG.AUDITLOG.LEVEL
    ESB 제품에 의한 Audit Logging 시 저장할 Pipeline Data Level을 설정한다.
    OFF : 로그를 저장하지 않음
    DEBUG : 테스트 및 디버깅을 위해서 많은 Pipeline Data를 저장한다.
    PROD : 시스템 운영에 꼭 필요한 Pipeline Data를 저장한다.
  • ESB.CONFIG.AUDITLOG.SMENUDISPLAY
    Socket Log > Local Server Audit Log 메뉴의 Display 여부를 설정한다.
    true : Display 함
    false : Display 하지 않음
    Audit Log 건수가 많을 경우 조회할 때 서버에 부하를 주기 때문에 Audit Log 건수에 따라서 Display 여부를 결정한다.
  • ESB.CONFIG.BUSINESSNM.LENGTH
    Basic Info > Business Name 메뉴에서 등록하는 Business Name의 길이를 설정한다.
  • ESB.CONFIG.BUSINESSNM.LENGTHCHK
    Basic Info > Business Name 메뉴에서 등록하는 Business Name의 길이를 고정길이로 할 것인지 여부를 설정한다.
  • ESB.CONFIG.COMPANY.NAME
    Enterprise 회사명을 설정한다.
    Space를 포함하지 않도록 한다.
  • ESB.CONFIG.CUSTOMMENU.CUSTOMER
    Customer에 따라서 Menu를 다르게 보이게 하기 위한 Customer의 Code를 설정한다.
  • ESB.CONFIG.DESIGN.PATH
    JSocketAdapter Administrator 화면 디자인 관련 파일의 기본위치를 설정한다.
    WmRoot : WmRoot Package에 있는 CSS, Images를 사용함
    custom : packages/JSocketAdapter/pub/custom 폴더에 있는 CSS, Images를 사용함
    JSocketAdapter Administrator의 화면 디자인은 기본적으로 WmRoot Package에 있는 CSS, Images를 사용한다.
    그러므로 운영중인 IS 버전의 디자인과 같게 된다. 그러나 운영중인 IS 버전이 10.3 보다 상위인 경우 JSocketAdapter를 개발한 IS 10.3 버전과 CSS, Images의 경로 및 파일명이 다르면 화면이 깨지게 될 것이다.
    이럴 경우에는 Basic Info > Custom Variable 메뉴의 ESB.CONFIG.DESIGN.PATH의 Value를 custom 으로 수정 후 화면을 Refresh 하면 packages/JSocketAdapter/pub/custom 폴더에 있는 CSS, Images가 적용된다.
    이 폴더에는 IS 10.3 WmRoot Package에 있는 CSS, Images가 있다.
  • ESB.CONFIG.EMAIL.HOST
    Email 서버의 Host Name 또는 IP를 설정한다.
    JSocketAdapter에서 수행하는 눈에 보이지 않는 기능을 수행 시 수행결과 또는 에러발생에 대한 내용을 ESB 관리자에게 메일로 통보를 할 때 사용할 Email 서버의 Host Name 또는 IP 이다.
    ESB 서버와 Email 서버와의 방화벽 오픈 등의 Network 환경을 고려해야 한다.
    Email 서버를 사용할 수 없는 상황일 경우에는 ESB.CONFIG.INCHARGE.EMAILYN의 Value를 반드시 false로 설정해야 한다.
  • ESB.CONFIG.EMAIL.HOSTPORT
    Email 서버의 Port 번호를 설정한다.
  • ESB.CONFIG.ERRORNOTI.EMAIL
    인터페이스 수행 시 에러가 발생했을 경우 에러 통보를 수신할 ESB 관리자의 이메일 주소를 설정한다.
    여러 개를 등록해야 하는 경우에는 '/' 를 구분자로 사용하고 첫 번째에 있는 이메일 주소가 보낸사람 및 받는사람 주소로 사용되고 두 번째 이후의 이메일 주소는 참조로 사용된다.
  • ESB.CONFIG.ERRORNOTI.EMAILYN
    인터페이스 수행 시 에러가 발생했을 경우 에러 내용을 ESB 관리자에게 메일로 통보를 할지 여부를 설정한다.
    true : 메일로 통보를 함, false : 메일로 통보하지 않음
  • ESB.CONFIG.ERRORNOTI.STEMAILYN
    인터페이스 수행 시 에러가 발생했을 경우 에러 내용을 Source System 담당자와 Target System 담당자에게 메일로 통보를 할지 여부를 설정한다.
    true : 메일로 통보를 함, false : 메일로 통보하지 않음
  • ESB.CONFIG.EXCEPT.IP
    Socket Server Port에 대한 접속을 제외시키는 IP를 설정한다.
    Socket Server Port를 Enable 시키면 L4 스위치에서 주기적으로 Health Check를 수행할 수 있다.
    운영서버가 두 대인 경우에 다른 운영서버에서도 Socket Server Port에 접속을 할 수 있다.
    해당 Port로 Client의 접속이 있을 때 수신되는 전문에 대해서 처리를 하는 Flow Service를 실행할 것인지 여부를 판단할 때 사용한다.
    Network Level에서 임의로 접속하는 IP가 존재할 경우에만 설정한다.
    L4 스위치가 아니더라도 Request를 Skip 해야 할 IP가 있는 경우에는 여기에 설정한다.
    운영서버인 경우 일반적으로 L4 스위치 IP, 운영서버 두 대의 IP를 모두 설정한다.
  • ESB.CONFIG.GWSERVER.ALIAS
    Socket Server에 대한 구성을 물리적으로 Gateway Server(Socket Port Open, Data Listening)와 Main Server(Data Processing)로 구분해서 운영하는 경우 Gateway Server에 대한 Remote Server Alias 목록을 설정한다.
    여러 개를 등록해야 하는 경우에는 '/' 를 구분자로 사용하고 IS Administrator > Settings > Remote Servers 메뉴에서 등록한 Alias를 입력해야 한다.
    운영서버가 두 대 이상이면서 L4가 있는 경우에는 모든 Gateway Server에 대한 Remote Server Alias를 등록할 필요 없이 L4에 대한 Alias 하나만 등록하면 된다.
    GW/MA 서버 구분이 있는 경우에 "ESB.CONFIG.GWSERVER.ALIAS"의 Value 대해서는 MA Server 끼리는 동기화 처리를 하고 GW Server에는 동기화 처리를 하지 않는다.
    Gateway Server에서는 설정할 필요 없고 Main Server에서 이 값을 설정해야 한다.
    Gateway Server에는 존재하면서 Main Server에는 존재하지 않는 정보를 Main Server에서 구하는 경우에 이 Alias가 사용된다.
    예) Local Server Socket 정보
  • ESB.CONFIG.INCHARGE.EMAIL
    ESB 관리자의 이메일 주소를 설정한다.
    여러 개를 등록해야 하는 경우에는 '/' 를 구분자로 사용하고 첫 번째에 있는 이메일 주소가 보낸사람 및 받는사람 주소로 사용되고 두 번째 이후의 이메일 주소는 참조로 사용된다.
  • ESB.CONFIG.INCHARGE.EMAILYN
    JSocketAdapter에서 수행하는 눈에 보이지 않는 기능을 수행 시 수행결과 또는 에러발생에 대한 내용을 ESB 관리자에게 메일로 통보를 할지 여부를 설정한다.
    true : 메일로 통보를 함, false : 메일로 통보하지 않음
    true로 설정하는 순간에 Server Resources Monitoring 및 Warning을 수행하는 데몬 서비스를 자동으로 실행시키고 false로 설정하는 순간에 실행 중인 데몬 서비스를 자동으로 종료시킨다.
  • ESB.CONFIG.INTERFACE.IDLENGTH
    Interface ID의 길이를 설정한다.
    Interface를 수행하는 메인 서비스 안에서 Namin Rule에 의해서 Interface ID를 추출해야 하는 경우가 생길 수 있다. 이와 같이 Interface ID를 추출할 때 사용한다.
  • ESB.CONFIG.LOCALFILE.CONFIGPATH
    ESB 서버 Local 파일 시스템에 쌓이는 파일의 주기적인 삭제를 위한 정보를 관리하는 XML 파일 저장위치를 설정한다.
    ESB 서버 Local 파일 시스템에는 Server Log, Socket Log 등 파일이 계속 늘어나서 파일 시스템의 사용 가능한 공간이 줄어들게 된다.
    JSocketAdapter에서는 일정기간 동안만 파일을 저장하고 일정기간이 지나면 파일 시스템에서 파일을 삭제하는 기능을 제공한다.
    이 Variable에 대해서는 Value를 수정할 수 없다.
  • ESB.CONFIG.MASERVER.ALIAS

    IS Administrator > Settings > Messaging > JMS Settings 메뉴에서 생성한 Connection Alias Name으로 설정한다.
  • ESB.SOCKET.TRIGGER.CONTSVC
    Custom Variable "ESB.SOCKET.ONLINE.TRANSTYPE"의 Value 변경에 따른 Trigger의 Enable 여부 변경 서비스를 설정한다.
    JMS로 변경 시에는 JMS Trigger의 Enabled는 True로 변경하고 webMethods Messaging Trigger의 Enabled는 False로 변경한다.
    MES로 변경 시에는 JMS Trigger의 Enabled는 False로 변경하고 webMethods Messaging Trigger의 Enabled는 True로 변경한다.
    해당 서비스는 시스템 상황에 맞게 개발하여야 하고 해당 서비스가 존재할 경우에만 Full Name으로 설정한다.
  • UM을 사용하지 않고 Java Queue를 이용해서 Socket 관련 로그를 저장하는 경우 Java Queue 생성 및 Queue 데이터 처리 데몬을 실행시키는 Variable
    ESB.SOCKET.LOG.SAVEYN ==> Local Server Log, Remote Server Log 두 개 생성 및 실행
    ESB.SOCKET.ONLINE.TRACESAVEYN ==> Server Trace, Client Trace 두 개 생성 및 실행
    ESB.SOCKET.ONLINE.TMPSAVEYN ==> 한 개 생성 및 실행
    ESB.SOCKET.MAILNOTI.SAVEYN ==> 한 개 생성 및 실행
    그러나, Java Queue를 사용할 경우 문제 발생의 소지가 있어서 2021.08.10일 이후로 Java Queue를 사용하지 않고 IS Local Queue를 사용하는 방식으로 변경됨.
